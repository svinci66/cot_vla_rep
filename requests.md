
# 需求说明书：基于 VILA-U 的 CoT-VLA 模型改造与复现

## 1. 需求背景与目标
*   **背景：** 现有的 VILA-U 模型具备多模态（图文视）的自回归生成与理解能力，但不具备机器人动作生成与物理世界闭环控制能力。
*   **目标：** 对 VILA-U 源码进行二次开发，引入视觉思维链（Visual CoT）、混合注意力机制（Hybrid Attention）和动作分块（Action Chunking），使其能够在 LIBERO 等仿真环境中执行基于语言指令的机器人操控任务。

## 2. 模型架构改造需求 (Architecture Modifications)

### 2.1 动作空间离散化与词表映射 (Tokenizer & Vocabulary)
*   **需求描述：** 大语言模型需输出离散的 Token，而机器人动作（7-DoF：空间坐标、旋转、夹爪）是连续数值。需构建连续动作到离散 Token 的双向映射。
*   **实现细节：**
    1.  **分箱 (Binning)：** 将每个动作维度的连续区间（取训练数据 1% 到 99% 的百分位数）均匀划分为 256 个离散的 Bin。
    2.  **词表重用：** 提取 VILA-U 文本分词器（Tokenizer）中**出现频率最低的 256 个 Token**。
    3.  **映射逻辑：** 在数据加载端（Dataloader），将真实动作数值映射为这 256 个 Token 的索引；在模型推理端，将输出的 Token 索引反向解码为物理执行的浮点数动作。

### 2.2 混合注意力机制实现 (Hybrid Attention Mask)
*   **需求描述：** 重写 Transformer 层的注意力掩码（Attention Mask）逻辑，打破全局因果掩码的限制。 
*   **实现细节：**
    1.  **前置阶段（因果掩码）：** 对于输入的[指令文本] + [当前观察图像] 以及需要生成的 [子目标图像 Token]，保持标准的下三角因果掩码（Causal Mask），确保自回归生成逻辑不变。
    2.  **后置阶段（全注意力掩码）：** 定位到序列末尾用于生成动作块（Action Chunk）的 Token 区域。将该区域的掩码矩阵修改为**全 1（即互相可见）**。
    3.  **跨区可见性：** 确保全注意力区域的动作 Token 能够“看到”前面的所有指令、当前图像和生成的子目标图像。

### 2.3 动作并行解码与占位符 (Action Chunking & Queries)
*   **需求描述：** 模型需要一次性（并行）输出一段长度为 $m$（例如 $m=10$）的动作序列，而不是逐个自回归生成。
*   **实现细节：**
    1.  **定义特殊 Token：** 在词表中引入预设的动作查询占位符（Action Queries），如代表平移的 `<x>`、旋转的 `<\theta>`、夹爪的 `<g>`。
    2.  **输入构造：** 在前向传播时，在序列末尾拼接入这些占位符 Token。
    3.  **输出对齐：** 使得模型在这些占位符位置的输出层（Logits），直接对应未来 $m$ 步的动作预测结果。

## 3. 训练逻辑改造需求 (Training & Optimization)

### 3.1 联合损失函数 (Joint Loss Function)
*   **需求描述：** 整合视觉生成与动作预测的优化目标。
*   **实现细节：**
    1.  **视觉损失 ($\mathcal{L}_{visual}$)：** 保留 VILA-U 原有的深度变换器（Depth Transformer）残差 Token 预测的交叉熵损失，用于约束子目标图像的生成质量。
    2.  **动作损失 ($\mathcal{L}_{action}$)：** 在动作查询占位符的位置，计算模型预测的动作 Token 与 Ground Truth（来自专家演示）之间的交叉熵损失。
    3.  **总损失：** 反向传播的总 Loss 为 $\mathcal{L} = \mathcal{L}_{action} + \mathcal{L}_{visual}$。

### 3.2 梯度与参数冻结策略 (Parameter Freezing)
*   **需求描述：** 保护模型原有的视觉特征提取能力，降低微调时的显存开销。
*   **实现细节：**
    1.  **冻结：** 遍历模型参数，将**视觉塔（Vision Tower）**的所有参数 `requires_grad` 设为 `False`。
    2.  **微调：** 仅放开 LLM 骨干网络（LLM Backbone）、投影仪（Projector）和深度变换器（Depth Transformer）的参数更新。

## 4. 数据管线改造需求 (Data Pipeline for LIBERO)

### 4.1 LIBERO 仿真数据预处理 (Data Preprocessing)
*   **需求描述：** 编写独立的数据清洗与加载脚本，将 LIBERO 的 `.hdf5` 格式转换为模型可接收的 Tensor 格式。
*   **实现细节：**
    1.  **去静止帧：** 计算相邻帧的动作差异，剔除差异低于阈值（机械臂未移动）的冗余帧。
    2.  **图像增强与对齐：** 读取环境图像，强制 Resize 至 $256 \times 256$ 分辨率；对所有图像应用 **180 度旋转**（校正 LIBERO 相机视角）。
    3.  **动作滑动窗口：** 将动作序列切分为长度为 $m$ 的片段（Chunk），与对应的起始帧图像进行绑定，构建 $(s_t, l, \hat{s}_{t+n}, a_{t \dots t+m})$ 的训练元组。

## 5. 验收标准 (Acceptance Criteria)

1.  **连通性测试：** 模型能够成功接收 LIBERO 预处理后的单条数据，且前向传播（Forward）不报错，能输出维度正确的动作 Logits。
2.  **掩码验证：** 打印或可视化 Attention Mask 矩阵，确认左上角为因果掩码，右下角动作区块为全 1 掩码。
3.  **收敛性测试：** 在少量 LIBERO 数据集（如 LIBERO-Spatial）上跑通微调流程（Backward），观察 $\mathcal{L}_{action}$ 和 $\mathcal{L}_{visual}$ 均呈稳步下降趋势。
4.  **闭环执行：** 将微调后的模型接入 LIBERO 仿真环境的 step 循环中，机械臂能够根据语言指令生成平滑的连续动作，而非剧烈抖动或崩溃。
